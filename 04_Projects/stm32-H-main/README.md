# stm32-H-main

基于 `STM32F103C8` 和标准外设库的智能小车工程。当前仓库中实际启用的是一个五路红外循迹版本，代码同时保留了编码器、PID、蓝牙串口、OLED、MPU6050、按键、普通串口等模块，适合作为 STM32 小车项目的综合练习工程。

## 项目概览

- 芯片型号：`STM32F103C8`
- 工程类型：`Keil MDK` 工程
- 库版本：`STM32F10x Standard Peripheral Library`
- 当前主功能：双电机差速循迹
- 工程文件：`project.uvprojx`

当前 `User/main.c` 中运行的逻辑非常直接：

1. 初始化编码器、小车驱动、电机、OLED、PWM、循迹模块。
2. 在 `while (1)` 中循环调用 `Track_Run()`。
3. 根据五路循迹传感器状态控制小车前进、左转、右转或停车。

需要注意的是，`main.c` 前半部分还保留了一整段“蓝牙 + PID 调速”的旧方案，但这部分已经被整体注释，当前不会参与编译运行。

## 当前功能

当前启用的循迹版本主要包含以下能力：

- 双路 PWM 电机控制
- 五路红外循迹传感器读取
- 小车基础动作封装：前进、左转、右转、停止
- 双路编码器初始化与读数接口
- OLED 模块初始化接口

其中，`Encoder_Init()` 和 `OLED_Init()` 虽然在 `main()` 中被调用了，但当前主循环并未使用编码器反馈闭环，也没有向 OLED 刷新内容。

## 核心源码说明

### 1. 主程序

`User/main.c` 当前入口逻辑如下：

- `Encoder_Init();`
- `Car_Init();`
- `Motor_Init();`
- `OLED_Init();`
- `PWM_Init();`
- `Track_Init();`
- 无限循环执行 `Track_Run();`

说明：

- `Car_Init()` 内部已经调用了 `Motor_Init()`，因此 `main()` 里的 `Motor_Init()` 属于重复初始化，通常不会出错，但有些冗余。
- 当前版本并没有做速度闭环控制，动作控制仍是固定 PWM 输出。

### 2. 小车运动控制

`Hardware/Car.c` 对电机动作做了简单封装：

- `Car_forward()`：双电机都设为 `40`
- `Car_Left()`：左轮 `40`，右轮 `30`
- `Car_Left1()`：左轮 `40`，右轮 `35`
- `Car_Right()`：左轮 `30`，右轮 `40`
- `Car_Right1()`：左轮 `35`，右轮 `40`
- `Car_Stop()`：双电机停转

这是典型的差速转向策略，便于快速验证循迹效果。

### 3. 循迹逻辑

`Hardware/Track.c` 使用五路传感器进行判断，命名方式为：

- `Left1`
- `Left2`
- `Mid`
- `Right1`
- `Right2`

当前代码中的动作判断大致如下：

- `00100`：前进
- 左侧检测到偏移：左转
- 右侧检测到偏移：右转
- `00000`：停车

这里的 `0/1` 具体对应黑线还是白底，要看你使用的红外模块输出极性；从现有判断来看，作者按“中间检测到线时前进”的方式进行了逻辑映射。

## 引脚分配

以下内容根据当前源码整理，接线前建议再次对照硬件实物确认。

### 电机与 PWM

- `TIM1_CH1 / PA8`：电机 PWM 通道 1
- `TIM1_CH2 / PA9`：电机 PWM 通道 2
- `PB0 / PB1`：电机 1 方向控制
- `PB10 / PB11`：电机 2 方向控制

### 编码器

- `TIM2 / PA0 + PA1`：编码器 1
- `TIM3 / PA6 + PA7`：编码器 2

### 循迹传感器

- `PA3`：`Left1`
- `PA2`：`Left2`
- `PA4`：`Mid`
- `PB4`：`Right1`
- `PB5`：`Right2`

### OLED

当前 OLED 驱动使用软件 I2C：

- `PB8`：`SCL`
- `PB9`：`SDA`

### 串口与蓝牙

- 普通串口 `USART1`
  - `PA9`：TX
  - `PA10`：RX
- 蓝牙串口 `USART2`
  - `PA2`：TX
  - `PA3`：RX

## 已知注意事项

### 1. 循迹引脚初始化有一处需要重点检查

`Hardware/Track.c` 中：

- `Read_Left2()` 读取的是 `PA2`
- 但 `Track_Init()` 在初始化 `GPIOA` 时配置的是 `PA3 | PA4`

这意味着 `Left2` 对应的 `PA2` 当前没有被正确初始化为输入上拉。若你发现左侧第二路循迹传感器读数异常，优先检查这里，并考虑把初始化引脚补为 `PA2 | PA3 | PA4`。

### 2. 预留模块之间存在引脚冲突

当前仓库中一些保留模块和循迹版本共享引脚，不能直接同时启用，例如：

- `Track` 使用了 `PA2 / PA3`
- `BlueSerial` 也使用了 `PA2 / PA3`
- `PWM` 使用了 `PA9`
- `Serial` 的 `USART1_TX` 也使用了 `PA9`

如果你准备把蓝牙调参、串口打印和循迹逻辑组合到一起，建议先统一梳理引脚复用方案。

### 3. 当前版本更偏“开环动作控制”

虽然仓库中已经有：

- 编码器读取
- PID 结构体与更新函数
- 蓝牙调参示例

但这些功能在当前启用的 `main.c` 中没有串起来，因此目前更像一个“基础循迹演示版”，而不是完整的速度闭环智能车版本。

## 后续可扩展方向

如果准备继续完善这个项目，可以考虑以下几个方向：

- 将编码器与 PID 接回主循环，做速度闭环
- 使用 OLED 显示传感器状态、编码器计数、目标速度等信息
- 接入蓝牙串口，实现在线调参
- 使用 MPU6050 做姿态估计或转向辅助
- 重构引脚分配，消除功能模块之间的冲突

## 适合谁

这个工程比较适合以下场景：

- STM32 标准库入门练习
- 智能小车模块化编程练习
- 差速电机控制与循迹实验
- 后续继续扩展 PID、蓝牙调参和传感器融合

如果你只是想先跑通一个最基础的循迹车，这个仓库已经具备核心结构；如果你想继续做成完整智能车，它也保留了不错的扩展基础。
