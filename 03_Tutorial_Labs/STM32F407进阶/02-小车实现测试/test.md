# 速度环复查分析

## 这次先给直接结论

结合你补充的 4 个现象，我现在更明确地判断：

1. `D-only` 没反应，这基本不是 bug，而是当前 `D` 实现的正常表现。
2. `P-only` 越接近目标越容易抖，主因不是 `P` 符号写反，而是当前速度环没有前馈/死区补偿，导致 `P` 必须同时负责“提供平均驱动力”和“纠偏”，为了减小稳态误差就不得不把 `Kp` 开大；一旦开大，又会撞上测速延迟和执行链路过于直接的问题。
3. `I-only` 能平滑接近目标，反而很能说明系统的主要矛盾是“需要一个稳定的平均 PWM 才能维持目标转速”，而不是“必须靠很强的瞬时纠偏”。
4. `PI` 存在时，`D` 调大只会加剧抖动，这和当前实现完全对得上：`D` 是直接对带平均窗口的测速值做差分，而且 PID 输出一旦变成负数，就会直接把电机反转驱动，这会把接近目标时的小波动放大成明显抖动。

所以这次更具体的判断是：

- 不是单纯“编码器有点波动”这么简单。
- 当前控制结构里，最关键的矛盾是：
  - 速度反馈有 80ms 量级的平均窗口/滞后
  - 没有死区补偿或前馈
  - PID 输出直接映射到双向 PWM
  - 输出一旦过零就直接反转电机

这几件事叠加，正好会得到你现在测到的现象。

## 先对照你的 4 个测试现象

### 1. 只调 P，离目标一半开始轻微抖，离目标四分之一开始剧烈抖

这个现象和当前代码是高度一致的。

关键点不在“P 有问题”，而在于 **现在没有任何基础驱动补偿**。

`PID_LoadSpeedConfig()` 里把 `outputOffset` 固定成了 `0.0f`：

- [`App/PID.c:42`](App/PID.c)
- [`App/PID.c:49`](App/PID.c)

也就是说，电机要维持某个稳定转速所需的平均 PWM，全都得靠误差本身去“顶出来”。

对 `P-only` 来说，控制律就是：

```c
Out = Kp * (Target - Actual)
```

这意味着：

- 如果目标转速下，电机实际需要一个非零平均 PWM 才能稳定转动
- 那么 `P-only` 必然要保留一个非零稳态误差，才能持续提供这个 PWM

可以写成：

```text
稳态误差 e_ss = 维持该转速所需平均PWM / Kp
```

所以现象会是：

- `Kp` 小时，不抖，但误差大，永远到不了目标附近
- 想把误差压小，只能继续加大 `Kp`
- `Kp` 一大，系统带宽上去，测速延迟和量化就开始显著放大
- 于是你会看到：离目标还比较远时能工作，一接近目标就开始来回抖

你提到“离目标一半开始轻微抖、离目标四分之一剧烈抖”，这反而很像典型的 `P-only` 被“稳态所需 PWM”逼着开太大的表现。

再结合你当前参数：

- 目标默认值 `300 rpm`：[`UserConfig/BoardConfig.h:340`](UserConfig/BoardConfig.h)
- 最大 PWM `1000`：[`UserConfig/BoardConfig.h:230`](UserConfig/BoardConfig.h)

如果 `Kp > 3.33`，那么刚起步时：

```text
u = Kp * 300 > 1000
```

就会直接打满输出。

而实际中，只要电机在 `300 rpm` 需要的平均 PWM 达到几百，想把稳态误差压到很小，`Kp` 基本就会逼近这个区间。  
这时“起步饱和 + 测速滞后 + 接近目标时过度纠偏”就会一起出现。

### 2. 关掉 P，只调 I，能很平滑地接近目标

这不是说 `I` 比 `P` 更“高级”，而是说明当前系统更需要的是“慢慢建立一个合适的平均 PWM”。

在当前实现里，积分项会逐步累加：

- [`App/PID.c:156`](App/PID.c)
- [`App/PID.c:165`](App/PID.c)

`I-only` 的效果本质上是：

- 一开始输出很小
- 然后慢慢积累
- 慢慢跨过电机死区/静摩擦
- 再慢慢逼近维持该速度所需的平均 PWM

这就是为什么它看起来“更平滑”。

换句话说，`I-only` 之所以表现好，不是因为系统特别适合积分，而是因为积分天然就是一个慢速爬升器，正好掩盖了下面这些问题：

- 电机死区
- 静摩擦
- 测速滞后
- P 太大时的瞬时过冲

所以这个现象很支持一个判断：

**系统主矛盾不是“纠偏不够快”，而是“缺少基础驱动量，P 为了补这个基础驱动量被迫开大”。**

### 3. P=0、I=0 时，只调 D 没反应

这个现象基本是正常的，不是 bug。

当前 D 项写法是：

```c
p->DOut = -p->Kd * (p->Actual - p->Actual1);
```

见 [`App/PID.c:166`](App/PID.c)

注意它是 **对测速值做差分**，不是对误差大小直接输出。

这意味着：

- 如果目标值固定不变
- 电机当前也没动
- 那么 `Actual == Actual1`
- `DOut == 0`

此时 `P=0`、`I=0`，整个控制器输出就是 0，电机当然不会自己启动。

所以 `D-only` 不能把一个静止电机拉到某个恒定目标转速，这在当前实现下是完全正常的。

更直接地说：

- `P` 负责“看当前差多少”
- `I` 负责“长期缺多少补多少”
- `D` 只负责“变化得太快时踩一脚”

如果前两者都没有，第三者本来就不可能单独把系统推起来。

### 4. PI 不为零时，D 调大只会加剧抖动

这个现象和代码也非常一致，而且这里我认为能定位到比较具体的原因。

原因不是单一一个，而是两个核心机制叠加：

1. `D` 看的不是“新鲜速度”，而是“20ms 采样 + 4 点滑动平均 + 整数 RPM”的测速值  
2. 只要 PID 输出变成负数，电机就会直接反转驱动

这两条叠加后，`D` 很容易把接近目标时的小波动，放大成正反向来回抽动。

## 关键代码点

### 1. 速度测量不是瞬时值，而是 4 点滑动平均值

编码器测速更新在：

- [`Hardware/Encoder.c:239`](Hardware/Encoder.c)

配置是：

- 编码器采样周期 `20ms`：[`UserConfig/BoardConfig.h:300`](UserConfig/BoardConfig.h)
- 平均点数 `4`：[`UserConfig/BoardConfig.h:306`](UserConfig/BoardConfig.h)
- 每圈计数 `1040`：[`UserConfig/BoardConfig.h:312`](UserConfig/BoardConfig.h)

测速公式本质上是：

```text
Speed_k = (delta_k + delta_{k-1} + delta_{k-2} + delta_{k-3}) / 4
```

所以你拿到的不是瞬时速度，而是近似 `80ms` 窗口上的平均速度。

这对 `P` 已经不是特别友好了，对 `D` 更糟。

### 2. D 项其实是在对“滑动平均后的速度”做差分

`D` 项在代码里用的是：

```text
Actual_k - Actual_{k-1}
```

但因为 `Actual` 本身已经是 4 点滑动平均，所以它的差分其实等价于：

```text
Actual_k - Actual_{k-1}
= (delta_k - delta_{k-4}) / 4
```

也就是说，当前 `D` 真正敏感的，不是“当前速度变化率有多平滑”，而更像是：

- 最新一个采样
- 和 80ms 前那个被窗口挤出去的采样

两者之间的差

这就会带来两个问题：

1. `D` 的有效信息已经被平均窗口推迟了  
2. `D` 会对窗口边界切换特别敏感

所以你把 `Kd` 调大后，看到的很容易不是“阻尼增强”，而是“窗口更新带来的差分脉冲被放大”。

这正好对应你说的：`D` 只会加剧抖动。

### 3. 更重要的一点：PID 输出过零后会直接反转电机

这个我认为是当前实现里最值得警惕的一点。

看 `Motor_ApplyDriveCommand()`：

- [`Hardware/Motor.c:141`](Hardware/Motor.c)

逻辑很直接：

- 输出大于 0：正转
- 输出小于 0：反转
- 输出等于 0：停

具体代码：

- 正转：[`Hardware/Motor.c:155`](Hardware/Motor.c)
- 反转：[`Hardware/Motor.c:161`](Hardware/Motor.c)

也就是说，对一个“固定正向速度目标”来说，只要 PID 算出来的值因为超调、D 项抑制、测速跳变等原因变成负数，系统就不是简单“减小驱动”，而是会：

- 立刻切到反向
- 给反向 PWM

这会导致什么？

- `P-only` 时，接近目标一旦略微超调，可能就开始给反向扭矩
- `PI + D` 时，D 项本来就在压加速度，`Kd` 稍大就更容易把输出拉过零
- 一旦过零，就从“减速”变成“反转驱动”

这会非常容易形成你感受到的“剧烈抖动”。

所以如果你现在是在给一个固定正速度设定值，物理上却允许控制器频繁打出负 PWM，那这个速度环天然就会很凶。

## 为什么 I-only 平滑，而 P/D 容易出问题

把你的现象翻译成控制意义，其实很清楚：

### I-only 平滑

因为它像一个慢速建立平均 PWM 的过程。

优点：

- 不会一下打太猛
- 不容易因为测速滞后立刻反向
- 更容易跨过电机死区后稳定在某个平均输出附近

缺点：

- 慢
- 理论上抗扰差
- 如果积分太大，后面仍可能过冲

### P-only 容易抖

因为它既要承担“纠偏”，又要承担“提供维持目标转速的平均驱动力”。

没有前馈/死区补偿时：

- `Kp` 小，驱动力不够
- `Kp` 大，接近目标就因为延迟和过零反转而振荡

### D 在这个速度环里不好用

因为当前 D 看的信号质量并不适合做微分。

更直白地说：

- 速度环本身测的已经是“速度”
- 你再对速度做微分，相当于在看“加速度味道”的量
- 这个量对编码器量化、窗口切换、机械间隙特别敏感

在很多直流电机速度环里，工程上更常见的是 **PI**，不是 **PID**。  
你现在测出来 “I-only 能慢慢逼近，D 只会让事情更糟”，其实和这个经验非常一致。

## 这次复查后，我认为最像问题的点

### 问题 1. 速度环没有基础输出补偿，P 被迫过大

证据：

- `outputOffset = 0.0f`：[`App/PID.c:49`](App/PID.c)

影响：

- `P-only` 必然保留稳态误差来换取平均 PWM
- 想减小误差，只能增大 `Kp`
- `Kp` 一大，系统就开始对测量延迟和波动过敏

这个点和你“越接近目标越抖”高度吻合。

### 问题 2. 输出一旦过零就直接反转，容易把小超调变成剧烈抖动

证据：

- [`Hardware/Motor.c:155`](Hardware/Motor.c)
- [`Hardware/Motor.c:161`](Hardware/Motor.c)

影响：

- 正向目标下，一旦输出被拉成负值，不是轻微减速，而是直接反向打扭矩
- `P` 大时会这样
- `D` 大时更容易这样

这是我现在最怀疑会把“轻微波动”放大成“明显机械抖动”的地方。

### 问题 3. D 项输入信号不适合做强微分

证据：

- 速度值来自 4 点滑动平均：[`Hardware/Encoder.c:253`](Hardware/Encoder.c)
- D 项直接用 `Actual - Actual1`：[`App/PID.c:166`](App/PID.c)

影响：

- `Kd` 增大后，更容易把测速窗口切换和量化台阶放大
- 对真正阻尼有用的信息不够“新鲜”

这和你“D 没阻尼效果，只有放大振荡”完全一致。

### 问题 4. 蓝牙在线调参不会重置 PID 内部状态

这个不是主因，但会影响你做“单项测试”的纯度。

蓝牙改 PID 后走的是：

- `Motor_RefreshPidProfiles()`：[`Hardware/Motor.c:472`](Hardware/Motor.c)
- `Motor_SyncPidProfiles()`：[`Hardware/Motor.c:187`](Hardware/Motor.c)
- `PID_ApplySpeedConfig()`：[`App/PID.c:119`](App/PID.c)

这里做的事情只是：

- 更新 `Kp/Ki/Kd`
- 更新限幅
- 对现有 `IOut/Out` 做 clamp

并没有调用 `PID_Reset()`。

所以如果你是在车还跑着的时候直接把：

- `P` 改成 0
- `I` 改成 0
- 或者切换不同组合

那么这次测试会多少带上一点上一组参数留下的内部状态。

它不是你现在现象的主因，但会让“单独测试 P/I/D”不够纯。

## 对你这次现象的最终解释

### 为什么 P-only 越接近目标越抖

因为 `P-only` 要靠误差本身来换取维持转速所需的平均 PWM。  
当你想把误差压小，就只能增大 `Kp`。  
一旦 `Kp` 够大，下面几件事就一起出现：

- 起步输出变猛，甚至饱和
- 反馈速度有 80ms 量级滞后
- 接近目标时，一点点超调就会让输出过零
- 过零后不是轻微减小 PWM，而是直接反转

所以它会从“还能追”很快变成“接近目标开始抽动”。

### 为什么 I-only 反而平滑

因为积分项会慢慢把平均 PWM 堆出来，不需要很大的瞬时增益。  
它天然更适合当前这种“电机需要基础驱动量、反馈又有滞后”的场景。

### 为什么 D-only 没反应

因为当前 D 只对测速变化起作用，不对固定误差直接起作用。  
静止起步时没有速度变化，所以没有输出，这个是正常行为。

### 为什么 D 加大只会更糟

因为当前 D 项：

- 对的是滑动平均后的速度
- 对窗口切换和量化很敏感
- 还会把输出更容易拉过零

一旦过零，电机就直接反转驱动，抖动自然明显。

## 这次我对“是不是编码器测速波动导致”的更精确回答

答案是：**是原因之一，但不是最关键的单独原因。**

更准确地说：

- 编码器测速的平均窗口和量化，确实让 D 项很难好用
- 但真正把它放大成明显机械抖动的，是“没有基础输出补偿 + 输出过零就反转”

如果只把锅甩给编码器，会漏掉更关键的控制结构问题。

## 最终判断

这次按你的现象重新对照后，我的判断比上次更具体：

- `D-only` 没反应：正常，不是 bug
- `I-only` 平滑：说明系统更缺“平均驱动力建立机制”
- `P-only` 接近目标就抖：主要因为没有前馈/死区补偿，`Kp` 被迫过大
- `D` 调大更抖：主要因为 D 对的是带平均窗口的测速值，而且输出过零会直接反转电机

如果只问“当前 PID 控制是不是有问题”，我的答案是：

**有，而且更像是控制结构层面的问题，不是某一行公式简单写错。**

当前最值得怀疑的两个核心点是：

1. `outputOffset` 为 0，导致 P-only 必须承担基础驱动量
2. PID 输出允许轻易过零并直接反向驱动电机，导致接近目标时特别容易抖
